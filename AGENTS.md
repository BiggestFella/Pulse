# AGENTS.md — Pulse

Agent-neutral guide for any AI coding agent (Antigravity, Cursor, Codex, Claude
Code, etc.) working in this repo. Claude Code users: `CLAUDE.md` has the same
conventions in more detail; this file is the cross-agent entry point.

## What this is
**Pulse** — a native iOS (SwiftUI, iOS 17+) solo workout tracker. Backend:
Supabase (Postgres). Project is **generated by XcodeGen** from `project.yml`.

## How to pick up a ticket
1. Tickets live in **Linear** (project "Pulse", team key `BAK`). You need a Linear
   integration/credentials configured — they are not in this repo.
2. Choose the **next available** ticket: status not Done **and not blocked**
   (Linear shows "blocked by" relations — never start a ticket with open
   blockers). If unsure, the only always-ready roots are **BAK-7** (Design System)
   and **BAK-6** (Data layer); everything else depends on them.
3. Move the ticket to **In Progress**.

## Where a ticket's instructions are (all in this repo)
- **Index:** `docs/superpowers/README.md` maps every `BAK-id → spec + plan` file.
- **Spec** (what to build, acceptance criteria): `docs/superpowers/specs/<date>-<slug>-spec.md`
- **Plan** (step-by-step implementation): `docs/superpowers/plans/<date>-<slug>-plan.md`
- **Product decisions (AUTHORITATIVE — overrides any spec "open question"):**
  `docs/superpowers/specs/2026-05-31-product-decisions.md`
- **Design source of truth:** `docs/design/` (screen specs, tokens, behavior).

The plan is self-contained: exact files, code, and test steps. Follow it.
(Plans mention a "REQUIRED SUB-SKILL" line that is Claude-specific — ignore it;
just execute the tasks.)

## Conventions (must follow)
- **SwiftUI, iOS 17+, MVVM + `@Observable`.** One screen = a View + an
  `@Observable` model in `Pulse/Features/<Feature>/`.
- **Never hardcode colors/spacing/fonts.** Use `Theme` tokens and the `PulseFont`
  typography API in `Pulse/Core/DesignSystem`.
- **Data access only through repository protocols** (`Pulse/Core/Data`). UI binds
  to those; mocks back them for UI-first work. Never call Supabase from a view/model.
- **Never hand-edit `Pulse.xcodeproj`** (it is git-ignored). Edit `project.yml`
  and run `xcodegen generate`.
- **TDD for logic** (models, analytics, engine): failing test → minimal code →
  passing test. **Views**: build + a `#Preview` + an XCUITest.

## Build & test
```bash
xcodegen generate
xcodebuild -project Pulse.xcodeproj -scheme Pulse \
  -destination 'platform=iOS Simulator,name=iPhone 17' test
```
Expected: `** TEST SUCCEEDED **`.

## Toolchain prerequisites (verify on your machine — NOT provided by the repo)
- **Xcode 26.x** (Swift 6.x) and the **iOS 26.5 simulator runtime** installed
  (the bundled SDK won't pair with older runtimes; install via
  `xcodebuild -downloadPlatform iOS` if a simulator destination is "ineligible").
- **XcodeGen** (`brew install xcodegen`).
- **GitHub auth** (`gh auth status`) for branch/push/PR.
- **Linear access** for reading/updating tickets.

## Git / PR workflow
1. Branch off `main`: `feature/<bak-id>-short-slug` (e.g. `feature/bak-6-data-layer`).
2. Conventional commits (`feat:`, `fix:`, `test:`, `docs:`, `chore:`). Commit often.
3. Tests must pass; `xcodegen generate` must run clean.
4. Open a PR (use the template), title it `<BAK-id>: <summary>`, body links the
   ticket (`Closes BAK-xx`). Move the ticket to **In Review**.
5. **STOP — do not merge.** A human reviews and merges. Merging unblocks the next
   tier of tickets.

## Don't
- Don't start blocked tickets, invent answers the design/decisions doc doesn't
  give (raise them instead), hardcode design values, hand-edit the `.xcodeproj`,
  or merge your own PR.
